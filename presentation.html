Este estudo especifico vai ter como foco o uso de modelos para time series forecasting e a sua performance em embedded systems. - Com objetivo, em perceber os trade-offs entre custo computacional e precisão dos modelos quando usados nestes dispositivos - Incidindo em particular em Transfer Learning e a sua aplicação em time series forecasting

Para ter alguma ideia do que esperar, é importante perceber o que já foi feito nesta área e para isso fiz alguma pesquisa sobre o que já foi feito em time series forecasting - O primeiro exemplo que tenho aqui é uma comparação entre LSTM e MLP, onde o LSTM teve melhor performance em todos os indicadores. Para o teste foram usados datasets de geração de energia solar. - O segundo exemplo analisa RNN tradicionais, LSTM e GRU. Os resultados variam consoante o dataset, não havendo um modelo que seja melhor em todos os casos. - Por último, temos uma avaliação de GRU vs LSTM and tanh-RNN. O GRU teve melhor performance em muitos casos, devido a exposição total do conteúdo da memória Principal conclusão: A escolha do modelo depende do dataset e do contexto do problema. Não havendo um claro vencedor quando falamos de modelos de machine learning para Time Series Forecasting.

Para trabalhos especificamente na área de Machine Learning em Embedded Systems analisei: - Transfer Learning, que revelou ser uma boa solução para reduzir o custo computacional enquanto mantém a performance. - Model Quantization, que tem como objetivo reduzir o tamanho do modelo e o tempo de inferência.

Para este estudo escolhi um dataset com dados de geração solar numa casa localizada no sul da Alemanha (por nenhuma razão especifica, apenas foi o dataset mais completo que encontrei). Com dados de 1 em 1 minutos Modelos: LSTM -> Modelo simples para comparar entre hardware numa fase inicial GRU -> Modelo mais complexo (com mais layers) para comparar entre hardware usando Transfer Learning

Para comparar entre hardware escolhi 3 sistemas: - Um computador de uso geral, com uma NVIDIA 1650 TI e 16GB RAM - Um Raspberry Pi 4 com 2GB RAM - Um Raspberry Pi 5 8GB RAM

Para conseguir avaliar bem as diferenças pretendidas realizei duas experiências distintas com os dois modelos: - Usei um modelo LSTM com poucas layers para garantir que conseguia replicar os resultados em todos os sistemas - Usei um modelo GRU com mais layers para comparar entre hardware usando Transfer Learning

Para a primeira experiência, o objetivo foi comparar a performance do modelo LSTM entre hardware. Ou seja, comparar o modelo treinado num computador de uso geral com o mesmo modelo treinado num Raspberry Pi 4 e num Raspberry Pi 5. Confirmar que os resultados são os mesmos em todos os sistemas para garantindo que são replicáveis. De formar a garantir que as diferenças que forem encontradas são devidas ao hardware e não a outros fatores.

Olhando para os resultados de performance do modelo é possivel verificar que os resultados são os mesmos em todos os sistemas, o que indica que o modelo foi replicado com sucesso. Estes 3 graficos apresentam os resultados de 3 métricas de performance nos 3 sistemas, e apresentam sempre o mesmo resultado ao longo do tempo de treino

As diferenças surgiram no tempo de treino e na custo computacional. Como vemos pelos resultados presentes nesta tabela, o tempo de treino no - RPi 5 foi ~50% maior que no computador de uso geral e - no RPi 4 foi ~304% maior.

- Em termos de custo computacional, os resultados são os esperados. O computador de uso geral tem um CPU mais potente e mais RAM, o que permite que o modelo seja treinado mais rapidamente e com menos custo computacional. - O Raspberry Pi 5 tem um CPU mais potente que o Raspberry Pi 4, o que permite que o modelo seja treinado mais rapidamente e com menos custo, mas com maior custo quando comparadao com o computador de uso geral. - O Raspberry Pi 4 tem um CPU menos potente e menos RAM, o que faz com que o modelo demore mais tempo a ser treinado e com mais custo computacional que os restantes sistemas.

- Para a segunda experiencia, o objetivo foi usar um modelo GRU treinado no computador de uso geral e usar Transfer Learning para retreinar o mesmo modelo nos Raspberry Pi 4 e 5. - Congelar as últimas quatro layers do modelo e treina-las nos Raspberry Pis. - Comparar os resultados para perceber se há perca de performance quando se usa Transfer Learning e analisar as diferenças de tempo de treino e poder computacional necessário.

- A laranja temos os resultados do modelo treinado de raiz e a azul os resultados do modelo treinado com Transfer Learning. - Como podemos ver pelos resultados, o uso de Transfer Learning não afeta a performance do modelo. Tendo conseguido atingir resultados semelhantes nas métricas de performance. - Com alguma variaçao, mas com um melhor tuning era possível atingir resultados bastante semelhantes numa situação real

Apesar da performance não ter sido afetada, o tempo de treino dos modelos reduziu bastante. Confirmando que é possível obter modelos com a mesma performance com menos custo computacional em embedded systems. - O tempo de treino no RPi 4 foi ~43% menor que o tempo de treino do modelo treinado de raiz - O tempo de treino no RPi 5 foi ~62% menor que o tempo de treino do modelo treinado de raiz

O uso do CPU e GPU também não foram excessão nessa redução. - O poder computacional necessário para treinar o modelo com transfer learning foi muito menor que o necessário para treinar o modelo de raiz. com valores por volta dos 20% quando ao treinar de raiz os valores eram por volta dos 60/70% - Já para não falar, que treina durante muito menos tempo, por isso também está a usar menos recursos durante menos tempo

As conclusões que tirei deste estudo foram: - É possível treinar um modelo de ML num embedded system, mas com um custo computacional significativo. - Para reduzir o custo computacional, é possível usar Transfer Learning, sem afetar a performance do modelo e reduzindo o overload computacional num sistema que tenha poucos recursos.

Como trabalho futuro, nomeadamente, para a continuação da minha dissertação: - Preciso de estudar melhor a aplicação de Transfer Learning em cenários reais, com datasets mais extensos e com mais variáveis - Explorar Model Quantization para reduzir o tamanho do modelo e o tempo de inferência, procurando sempre um balanço entre performance e custo computacional - Explorar mais técnicas que possam reduzir o custo computacional enquanto mantém a performance, como por exemplo Model Pruning
